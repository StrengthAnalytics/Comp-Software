# Changelog

All notable changes to Comp-Software are documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/), and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html): `major.minor.patch`.

- **Major**: breaking changes to the data model, public API, or operator-facing workflows
- **Minor**: new functionality, additive schema changes
- **Patch**: bug fixes, copy changes, internal refactors

Update the `[Unreleased]` section in every PR. Cut a new version entry when deploying to production.

## [Unreleased]

### Changed
- Structured squat/bench rack settings on entries (replacing the two free-text rack-height columns). Squats now store an integer rack height plus a rack setting (IN / OUT / LEFT IN / RIGHT IN); bench stores an integer bench height, an integer safety height, and a spotting choice (SELF / HAND OUT). New `squat_rack_setting` and `bench_spotting` Postgres enums and reworked `entries` columns in migration `20260527000001_weigh_in_rack_settings.sql` (the old text columns are dropped and recreated as integers — no rack data preserved), with `types/database.types.ts` hand-updated to match. Editable on both the weigh-in and entries screens; the `/run` scoresheet (currently a stub) can capture them later once built. Enum labels/values and the rack-height/enum Zod fields added to `lib/constants.ts` and `types/entry.ts` (`entryUpdateSchema` and `weighInSchema`)
- Surnames are now optional for lifters (mononymous athletes and sparse imported data). The first name stays required, so every lifter still has a name; a blank surname is stored as an empty string. Relaxed in the lifter Zod schema, the bulk-import parser, and the add/edit forms. Names now render via a shared `formatLifterName` helper (`lib/lifters/name.ts`) that shows just the first name when the surname is blank, across the entries, flights and teams screens

### Changed
- The sessions & flights screen now assigns by team for team competitions instead of by lifter: the roster board shows team cards (with their members) that move into flights as a unit (every member's entry moves together, optimistically), via `assignTeamFlightAction`. When no teams exist yet it prompts the operator to create teams first, linking to the Teams screen. Non-team comps keep the per-lifter roster board

### Added
- Dedicated weigh-in screen at `/(admin)/[comp-slug]/weigh-in`, separate from the inline editing on the entries screen. The operator picks a session (weigh-ins run per session) and works down the lifters in calling order: non-team comps group by sex (female first, then male) and team comps group by lift role (squat, bench, deadlift) then sex, with each group ordered by flight then lot. Each lifter card captures bodyweight, openers and rack/bench settings; "Save & mark weighed in" needs only bodyweight and the contested openers (rack details can be filled in later), and a "Save progress" button records a partial card without flipping the status. Team members show only their assigned lift's opener. Single station, so it reads from the server and refreshes on save rather than subscribing to real-time. Shows a per-session weighed-in count and flags any registered lifters not yet assigned to a flight. Pure ordering helper `buildWeighInGroups` in `lib/weigh-in/order.ts` (unit-tested), `weighInAction` in `actions/entries.ts` (admin-guarded, Sentry-wrapped, validated against the new `weighInSchema` — a subset of the entry update so it can't clobber weight class / division / lot), UI in `components/weigh-in/weigh-in-manager.tsx`. Reachable from a "Weigh-in →" link on the competition edit screen
- Weigh-in workflow aids on that screen: weighed-in lifters sink to the bottom of their group and collapse to a compact green row (with Edit/Collapse to reopen for corrections); a per-session name filter to jump to a lifter; and a weight-class check — each card shows the lifter's class as a reassignable select and an amber warning when the recorded bodyweight falls outside it (or no class is set), suggesting the class the bodyweight fits. Pure helpers `isBodyweightInClass` / `findWeightClassForBodyweight` in `lib/weigh-in/weight-class.ts` (unit-tested; IPF exclusive-lower/inclusive-upper bounds), and `assignEntryWeightClassAction` in `actions/entries.ts` (admin-guarded, Sentry-wrapped, validates the class belongs to the comp and matches the lifter's gender) with `assignWeightClassSchema` in `types/entry.ts`
- Team standings on the public results page (`/(public)/[comp-slug]/results`) for team competitions: teams ranked by the sum of their three members' IPF GL points, each taken from the member's best successful lift, with a per-team member breakdown (best lift and points). Pure `computeTeamStandings` in `lib/scoring/team-standings.ts` (100% covered, standard tie-aware ranking); the page reads names and gender from the PII-free `public_lifters` view so it works for anonymous visitors, and best lifts from `good_lift` attempts. Until the comp is published it shows a notice instead of an empty table (the view exposes names only for publicly visible comps). A "Team standings →" link is added to the competition edit screen for team comps
- Bulk add teams on the teams screen: a collapsible card that creates many teams at once from a pasted list of names (one per line). Skips names that already exist in the comp (case-insensitive) and duplicates within the paste, with a per-name summary. Pure parser `parseTeamNames` in `lib/teams/bulk-add.ts` (unit-tested), `bulkAddTeamsAction` in `actions/teams.ts`, UI in `components/teams/bulk-add-teams.tsx`
- "Delete all entrants" on the entries screen: a danger-zone button behind a custom type-to-confirm dialog (the operator types the competition name to enable it) that removes every entry in the comp in one go, cascading to their attempts and referee decisions. The persistent lifter (person) records are kept and can be re-registered. Blocked once the comp is completed, since the delete cascades to attempts and results. `deleteAllEntriesAction` in `actions/entries.ts` (admin-guarded, Sentry-wrapped) returns the number removed; UI in `components/entries/delete-all-entries.tsx`
- Team competition format. A `Team competition` toggle on the competition details (shown for full-power comps only) flags a comp where each team is three lifters — one each on squat, bench and deadlift — and the team score is the sum of their IPF GL points. New teams management screen at `/(admin)/[comp-slug]/teams`: name teams and fill each team's three role slots from the pool of registered lifters (optimistic assignment, rolled back on failure), with an unassigned-lifters list. Reachable from a link on the competition edit screen that appears only when the comp is a team competition
- Server actions for teams (`actions/teams.ts`: create/update/delete plus `assignEntryTeamAction`): each runs `adminGuard()`, validates with Zod, is wrapped in `Sentry.withServerActionInstrumentation`, and returns a typed `ActionResult`. Assignment verifies the entry and target team belong to the comp and maps the one-member-per-lift unique violation to a friendly message; team deletion unassigns members first so the entry role check stays satisfied
- Team Zod schemas in `types/team.ts` (team create/update, assignment with a both-or-neither team/lift rule), `is_team_competition` on the competition schema (gated to full-power comps), and `LIFT_LABELS` in `lib/constants.ts`
- Unit tests for the team schemas (`tests/unit/team/schema.test.ts`) and the team-competition rules on the competition schema
- IPF GL Points scoring engine in `lib/scoring/ipf-gl.ts`: pure `ipfGlPoints` using the official IPF 2020 coefficients (full-power classic/equipped × sex) and `teamGlScore`, which sums a team's three members' GL points taken from each member's best lift. Groundwork for the team-competition format: the full-power (3-lift) coefficients are applied to every role, including the single-squat and single-deadlift members, because the IPF publishes no official single-squat or single-deadlift GL set. 100% unit coverage in `tests/unit/scoring/ipf-gl.test.ts`, anchored by an end-to-end oracle from the official worked example (male classic, 82.5 kg, 550 kg → 76.37)
- Sessions & flights management at `/(admin)/[comp-slug]/flights`: a single screen to build the meet's running structure and assign lifters to flights. Sessions (name, date, start time, optional platform, sort order) each contain flights; below the structure editor a roster board shows a lane per flight grouped by session plus an "Unassigned" lane, with lifters moved between flights via a per-lifter dropdown (optimistic local update, rolled back on failure). Within each lane lifters preview in running order (first contested lift's opener ascending, then lot number) via the pure `compareFlightOrder` helper in `lib/flights/order.ts` (unit-tested). Flight lanes warn past `MAX_FLIGHT_SIZE` (14). Single-platform by default — the per-session platform picker appears only once a second platform is added. Reachable from a link on the competition edit screen
- Server actions for the meet structure (`actions/platforms.ts`, `actions/sessions.ts`, `actions/flights.ts`: create/update/delete each) plus `assignEntryFlightAction` in `actions/entries.ts`: each runs `adminGuard()`, validates with Zod, is wrapped in `Sentry.withServerActionInstrumentation`, and returns a typed `ActionResult`. Cross-entity rules RLS cannot enforce are checked in the action (a session's platform and a flight's session must belong to the same comp; a move's target flight must belong to the comp). Deleting a session or flight that still holds lifters is blocked with a friendly message rather than silently unassigning them
- Session, flight and platform Zod schemas in `types/flight.ts`, and a `MAX_FLIGHT_SIZE` constant in `lib/constants.ts`
- Unit tests for the flight running-order comparator (`tests/unit/flights/order.test.ts`)
- "Copy current entries" export on the entries screen: copies the comp's registrations to the clipboard as tab-separated rows using the exact same columns (and event-type-driven openers) as the bulk import, so they paste straight into a sheet, can be edited, and re-imported. Pure formatter `formatBulkExport` in `lib/entries/bulk-import.ts`, round-trip unit-tested against the parser
- Bulk lifter import on the entries screen: a collapsible "Bulk import" card that copies a tab-separated header row (so the operator can paste it into a Google Sheet, fill a row per lifter, and paste the cells back) and parses the pasted block into validated rows with a live preview. Columns follow the comp's event type (openers only for contested lifts). Headers: First name, Surname, Gender, Date of birth, Membership number, Club, Country, Division, Weight class, Lot, Bodyweight, and per-lift openers. Gender accepts M/F or Male/Female; dates accept ISO or DD/MM/YYYY; division and weight class are matched by name (unmatched names are warnings, not failures). Existing lifters are matched by surname + first name and reused (details overwritten); anyone already registered in the comp is skipped. Returns a per-row summary (created / updated / skipped / errors). The single "New lifter" add remains for one-off changes. Pure parser in `lib/entries/bulk-import.ts` (unit-tested), `bulkImportEntriesAction` in `actions/entries.ts`, UI in `components/entries/bulk-import.tsx`
- Lifter registration & weigh-in at `/(admin)/[comp-slug]/entries`: a single screen to add lifters to a comp and edit their weigh-in details inline. Add by surname search of existing lifters (membership numbers change yearly, so surname is the search key) or create a new lifter; each registered entry is an editable card for weight class, division, lot number, bodyweight, per-lift openers, rack heights, and status. Opener and rack fields follow the comp's event type (full power / bench only / deadlift only). Reachable from a link on the competition edit screen
- Server actions for registration (`actions/lifters.ts`: `searchLiftersAction`, `createLifterAction`, `updateLifterAction`; `actions/entries.ts`: `createEntryAction`, `updateEntryAction`, `deleteEntryAction`): each runs `adminGuard()`, validates with Zod, is wrapped in `Sentry.withServerActionInstrumentation`, and returns a typed `ActionResult`. Entry updates validate cross-entity rules RLS cannot (weight class and division must belong to the comp; weight class gender must match the lifter) and map the unique constraints to friendly messages (already-registered, lot number taken). Per the operator decision, entry writes are not gated on competition status — all comps stay editable
- Entry Zod schemas in `types/entry.ts` (lifter create/update/search, entry create/update) and entry-status labels plus an event-type→lifts map (`ENTRY_STATUS_LABELS`, `ENTRY_STATUSES`, `LIFTS_FOR_EVENT`) in `lib/constants.ts`
- Unit tests for the lifter and entry Zod schemas (`tests/unit/entry/schema.test.ts`)
- Admin sign-in at `/auth`: email + password form (`components/auth/sign-in-form.tsx`) backed by `actions/auth.ts` (`signIn`, `signOut`), restricted to `ADMIN_EMAILS`. Initial-build auth choice — production switches to 6-digit OTP (see ARCHITECTURE.md §7); the `ADMIN_EMAILS` allowlist is the authorization gate regardless of sign-in method
- Route protection: `proxy.ts` redirects unauthenticated visitors away from `/comps`, and the `(admin)` layout is now an async server component that gates on `isAdmin()` and renders the admin chrome (nav + signed-in email + sign-out)
- Competition setup UI under `/comps`: list, create (`/comps/new`), and edit (`/comps/[id]/edit`) screens. The edit screen manages the comp plus its divisions and weight classes, including a "Seed IPF defaults" button for the standard classic divisions and weight classes
- Server actions for competition setup (`actions/competitions.ts`, `actions/divisions.ts`, `actions/weight-classes.ts`): each calls `requireAdmin()` via the `adminGuard()` helper, validates with Zod, is wrapped in `Sentry.withServerActionInstrumentation`, and returns a typed `ActionResult` rather than leaking raw DB errors (unique-violation mapped to friendly messages)
- Shared Zod schemas and a `slugify` helper in `/types` (`competition.ts`, `auth.ts`), an `ActionResult` discriminated union (`types/action-result.ts`), `toFieldErrors` (`lib/validation.ts`), and IPF default divisions/weight-class data plus enum labels in `lib/constants.ts`
- Unit tests for the competition and auth Zod schemas (`tests/unit/competition`, `tests/unit/auth/auth-schema.test.ts`)
- Database foundation: sequential migrations in `/supabase/migrations` for the full data model (profiles, competitions, divisions, weight_classes, platforms, sessions, flights, lifters, entries, attempts, referee_decisions) plus the eight domain enums (`kit_type`, `event_type`, `comp_status`, `entry_status`, `lift_type`, `attempt_result`, `ref_position`, `ref_decision`)
- Row Level Security on every table: admins (any authenticated session) read and write everything; anon reads data belonging to publicly visible competitions, gated by SECURITY DEFINER helpers (`is_comp_public`, `lifter_in_public_comp`)
- `public_lifters` view exposing only non-PII lifter fields (name, gender, club, country) to anon, scoped to lifters in a publicly visible competition; the base `lifters` table is admin-only (full PII)
- Logical replication enabled (`supabase_realtime` publication + `replica identity full`) on `attempts`, `referee_decisions`, `entries`, `flights`, `sessions`
- `lib/auth/admin.ts` admin allowlist (`isAdmin`, `requireAdmin`) checked against the `ADMIN_EMAILS` env var, with typed `AuthorizationError`; 100% unit coverage in `tests/unit/auth`

### Changed
- Simplified the auth model to an admin allowlist (`ADMIN_EMAILS`) plus anonymous public read of published comps. Removed the role-based permission matrix, the `comp_roles` table, the `comp_role` enum, `requireRole()`, the role helpers (`has_comp_role`, `has_any_comp_role`), the creator-grants-meet_director trigger, and the per-comp `overlay_key` and `created_by` columns. Overlays now run on the admin session (no separate overlay auth). See the ADR in ARCHITECTURE.md section 7. Writes rely on `requireAdmin()` in server actions as the sole gate, which requires public sign-ups to stay disabled.
- Public read on `lifters` exposes only non-PII fields: anon reads the `public_lifters` view (name, gender, club, country — no date of birth, no IPF member ID), scoped to lifters in a publicly visible comp; the base table is admin-only.
- Documentation updated to reflect online-only, hosted-first development workflow (no local Next.js or Supabase; migrations applied via the Supabase SQL editor; `types/database.types.ts` hand-authored to match)
- The `@sentry/cli` install build script is currently unapproved (pnpm `onlyBuiltDependencies`). It must be approved before the first production deploy, since Sentry source-map upload runs at build time and depends on the `sentry-cli` binary.
- Email delivery: Supabase default SMTP service used in development until a production domain is registered. Resend integration (per CLAUDE.md and ARCHITECTURE.md) deferred until then. No code changes required at switchover; only Supabase Auth SMTP settings.
