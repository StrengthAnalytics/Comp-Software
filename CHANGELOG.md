# Changelog

All notable changes to Comp-Software are documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/), and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html): `major.minor.patch`.

- **Major**: breaking changes to the data model, public API, or operator-facing workflows
- **Minor**: new functionality, additive schema changes
- **Patch**: bug fixes, copy changes, internal refactors

Update the `[Unreleased]` section in every PR. Cut a new version entry when deploying to production.

## [Unreleased]

### Added
- **Team placings on the run screen and warm-up board.** For a team competition the **Current place** and **Predicted place** columns now show the lifter's *team's* place — ranked against the other teams by team GL (actual and projected) — on every member's row, rather than an individual lift placing (a team comp has no individual placing). They sit alongside the existing Team points / Predicted team points columns, all togglable. The team ranking comes from the same `computeTeamStandings` the public results page uses (extended with a `predictedRank`), bridged from the live board via a new `computeBoardTeamStandings` helper (`lib/scorekeeper/team-board-standings.ts`, replacing the previous `team-points.ts`), so the boards and the public page can never disagree on a team's points or place. A team with no points yet shows a dash. Individual comps are unchanged.
- **Projected totals on the public team-standings page** (`/(public)/[comp-slug]/results`). Each team now shows its **projected total** beneath its actual total (in amber, only when higher), and a member with a heavier declared-but-unjudged attempt shows that projected lift/points under their current figure — so a spectator can see where a team would land if the current attempts are made, the same "if they make what's declared" projection the run/warm-up boards use. The list stays ranked by the actual total. The page now seeds *all* attempts (not just good lifts) so the prediction has the in-play attempts, and `computeTeamStandings` (`lib/scoring/team-standings.ts`) returns a `predictedTotal` per team plus `predictedBestLiftKg`/`predictedPoints` per member, summed through the same `teamGlScore`. `useTeamStandings` now derives both the actual and predicted member bests via the shared `bestLiftFor`/`predictedBestLiftFor` helpers (replacing its hand-rolled best-good-lift loop), so the public page can't drift from the boards. (The hook's `initialBestAttempts` prop is renamed `initialAttempts` to reflect that it seeds the full set.)
- **Standings columns on the run screen and warm-up board** (each togglable from the **Options ▾** dropdown, remembered per browser, default off). For an individual competition: **Current place** (the lifter's live placing within their category, from the running total achieved so far), **Predicted place** (placing from the projected total), **Predicted total** and **Predicted GL points**. The prediction is the GoodLift/LiftingCast "if they make what's declared" projection — each lift's predicted best is the heaviest of that lift's attempts that are either already a good lift *or* still pending with a declared weight (a missed attempt drops out), so a declared-but-untaken third deadlift counts toward the predicted total until it is judged, and a squatter's predicted total already includes their declared bench and deadlift openers. A predicted total is only counted once every contributing lift has a value in play (a bombed lift with nothing pending means no projected total), matching IPF's must-total-in-every-lift rule. Placement category is weight class × division × sex (kit type is fixed per comp); a lifter with no class/division, or no total yet, shows a dash. Ties break by lighter bodyweight then lower lot (the IPF order). For a **team competition** the four individual columns are replaced by **Team points** and **Predicted team points** — the lifter's team's current and projected summed IPF GL (each member's best good lift, or predicted best), shown on every member's row — since a team comp has no individual placing. All values reconcile live through the existing attempts/entries subscriptions (no refresh), and the places rank the whole field comp-wide even though the warm-up board only shows one platform. New pure, unit-tested helpers single-source the maths so the run screen and warm-up board can't disagree: `predictedBestLiftFor`/`computePredictedScore` (`lib/scorekeeper/entry-score.ts`), `computePlacings` (`lib/scorekeeper/placings.ts`) and `computeTeamPoints` (`lib/scorekeeper/team-points.ts`, summing through the shared `teamGlScore`). `BoardEntry` now also carries `weightClassId`, `divisionId` and `teamId` (already fetched by `loadBoardData`) for the category grouping and team aggregation. The weigh-in screen is intentionally unchanged — it has no results yet, only openers being captured.
- **Broadcast overlays (OBS) panel** on the competition Setup screen (`/comps/[id]/edit`): a copy-URL control for the OBS overlay URLs (currently the lifter lower-third), since an overlay is an OBS Browser Source rather than a page an operator visits. It offers a per-platform URL (`?platform=`) when more than one platform has a session, otherwise the bare overlay URL (which auto-selects), each with a **Copy URL** button (built against the live origin, so it targets whichever preview/production deployment the operator is on) and a **Preview** link that opens the overlay in a new tab. UI in `components/comps/overlay-links.tsx`; the edit page now also loads the comp's platforms/sessions to drive it.
- **Lifter lower-third overlay** (`/(overlay)/[comp-slug]/lifter`), the first real OBS broadcast overlay (the route was a stub). A LiftingCast-style transparent lower-third for the lifter **currently on the platform**: their name, category (weight class · division, or the team name in a team comp), the attempt being lifted (flight · lift · attempt number) with the declared weight in kg (and an informational lbs figure), and a best-lifts/total strip (best squat/bench/deadlift, total, and IPF GL points once they total). It sits in the chrome-less `(overlay)` route group on a transparent background and renders a **fixed 1920×1080 canvas** — only the card paints, so an OBS **Browser Source carries the page's alpha natively and no chroma key is needed** (set the source URL to the deployment's `/<comp-slug>/lifter?platform=<id>`, 1920×1080). Each new lifter/attempt re-mounts the card on its natural key so it slides up fresh (a new `animate-overlay-rise` utility registered in `app/globals.css` via Tailwind v4's `@theme`). Like the public warm-up board it reads **anon, RLS-gated** (`loadBoardData({ publicView: true })` sources names from the PII-free `public_lifters` view; every other read is covered by the publicly-visible-comp anon policies), so it works in OBS's headless browser — which does not share the admin session — once the comp is published, with no separate overlay auth. It shows an "appears once published" notice for a still-draft comp and a platform chooser when several platforms have sessions and none is chosen (`?platform=`), reusing the display chooser. The current lifter is **derived, not stored**: it reconciles live through the shared `useBoardState` (attempts/entries/flights, scoped by `competition_id`) and picks the on-platform lifter via the new shared `buildPlatformLiveView` helper, so the overlay can never disagree with the run screen or warm-up board about who is up. ### Changed
- **Public team standings now update live** (`/(public)/[comp-slug]/results`). The page was server-rendered once, so during an active meet a viewer had to refresh to see a team's score move — at odds with the real-time-first mandate, and the page is explicitly viewable while the comp is `active`. It now seeds from the server and hands off to a client component (`components/results/team-standings-live.tsx`) that subscribes to `attempts` and `entries` for the competition (the shared `/lib/realtime` hooks, scoped by `competition_id`) and re-ranks on every change, so new good lifts, weight corrections, result flips and weigh-in bodyweights all re-score teams without a reload. A small live/reconnecting/offline pill (the shared `computeConnectionIndicator`) shows connection health. The reconcile is a new composed hook `useTeamStandings` (`lib/realtime/use-team-standings.ts`) reusing the existing `applyAttemptChange`/`attemptKey` for the attempts map plus a new pure `applyTeamMemberChange` (unit-tested, `tests/unit/realtime/team-member-change.test.ts`) that folds entry changes into the members map, preserving each lifter's name/sex from the PII-free `public_lifters` load (the entry realtime row can't carry them). Teams themselves have no realtime channel (the `teams` table has no logical replication), so a team rename or a brand-new team still appears only on the next load — both rare mid-meet, and an empty new team scores 0 regardless. ARCHITECTURE.md §4 real-time map updated.
- **Team standings list members in lift order — squat, then bench, then deadlift, top to bottom** — regardless of the order the data was gathered in. `computeTeamStandings` (`lib/scoring/team-standings.ts`) now sorts each team's members by `TEAM_LIFTS` order before returning them, so every team card reads consistently. Covered by a new ordering test.
- Single-sourced the team-GL sum: `computeTeamStandings` now derives a team's total via `teamGlScore` (`lib/scoring/ipf-gl.ts`) rather than re-implementing the sum-and-round, so the public standings, the run-screen GL column and the overlays can never aggregate or round a team differently (`teamGlScore` was previously only exercised by tests). The duplicated local `round2` in `team-standings.ts` is gone with it.
- **Team comps group the run and warm-up boards by lift, not by flight.** In a team competition each member contests a single assigned lift, so the live roster on the run screen and the warm-up board now groups the whole session by lift — every squatter together, then every bencher, then every deadlifter — instead of by each flight's single current lift (which clustered a flight's squatter, bencher and deadlifter together and pushed the next flight's squatters far down the board). Within a lift the flight currently working it leads, flights yet to reach it follow in flight order, and a flight that has finished that lift (its third round decided) drops below the active ones; a flight merely paused between rounds stays put, and within an active flight the round-in-progress bar weight still orders the lifters (lightest first). Individual comps are unchanged. The team ordering lives in a new `orderTeamSessionRoster` helper in `lib/attempts/running-order.ts`; it shares the per-flight/per-lift state accumulation (`buildFlightLiftStates`), the lift-complete test (`isFlightLiftComplete` — so the two paths agree on when a flight has left a lift and a between-rounds gap never reads as finished) and the inactive tie-break (`compareInactiveRoster`) with the individual `orderSessionRoster`. Both the run screen (`scoresheet-board.tsx`) and the warm-up board (`warm-up-display.tsx`) order through one shared `orderRosterForSession` helper (`lib/scorekeeper/order-roster.ts`) that owns the roster projection and the team-vs-individual choice, so the two screens can't drift.
- The rack-heights screen now updates in real time: it subscribes to `entries` and `flights` changes for the competition (via the shared `/lib/realtime` hooks, scoped by `competition_id`) and coalesces a server re-pull, so a second device — the head table, or another warm-up phone — sees roster changes, flight assignments and racks-set completion live. A row's in-progress edits are local state seeded at mount, so the refresh re-orders and re-collapses the list without clobbering what the operator is typing. ARCHITECTURE.md §4 real-time map updated. Known limitation: individual rack *field values* are local-authoritative per device — a value typed on one device doesn't live-update an already-open row on another device; it appears when that row next loads (a reload, or the row re-rendering from a roster change).
