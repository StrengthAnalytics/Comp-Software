# Changelog

All notable changes to Comp-Software are documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/), and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html): `major.minor.patch`.

- **Major**: breaking changes to the data model, public API, or operator-facing workflows
- **Minor**: new functionality, additive schema changes
- **Patch**: bug fixes, copy changes, internal refactors

Update the `[Unreleased]` section in every PR. Cut a new version entry when deploying to production.

## [Unreleased]

### Fixed
- Code-review fixes on the new live running order (`/run`):
  - The roster no longer jumps to the next lift's order during the gap between rounds. Because every lift's opener is seeded pending at weigh-in, the lead-lift detection used to flip to bench the moment squat round 1 resolved before round 2 was declared; `leadRoundByFlight` now treats a lift as still in progress until its final round is reached or a later lift has actually started lifting, and holds the last round's order through the gap. Added regression tests for the gap and for advancing once the next lift starts.
  - Scoresheet gridlines switched from `border-collapse` to a `border-separate` model (per-cell right/bottom borders, top on the header row, left on the frozen column) so the bold black lines stay attached to the sticky header and frozen lifter column when the table scrolls, instead of dropping off as collapsed borders do on sticky cells.
- Pre-merge review fixes on the scorekeeper grid:
  - Re-saving a weigh-in no longer reverts a platform-side correction to a first attempt: editing attempt 1's weight on `/run` now writes back to the entry's opener column (`setAttemptWeightAction`), so the opener and attempt 1 stay in step and the weigh-in re-seed is a no-op rather than a clobber.
  - The ✓ / ✗ result buttons are disabled until an optimistically-declared attempt has its real id, so a fast click can no longer send a `temp:` placeholder id that the result action rejects.
  - The live session and roster are now derived from session/flight structure instead of only from existing attempts: a flight of weighed-in lifters shows before any attempt is declared, and the board no longer jumps sessions during a between-rounds gap (a session is "finished" only once a later session has started lifting).
  - The board now subscribes to flight changes (`useFlightsSubscription`) so flight reorders/reassignments reflect live, and re-seeds its state from fresh server props on reload so a refresh recovers from any missed real-time events.
- Shared `nullsLast` / `compareValues` sort helpers extracted to `lib/ordering.ts` (deduped from the flights, weigh-in and running-order comparators), unit-tested. Removed the now-unused `ATTEMPT_RESULT_LABELS` constant; the Total cell computes the per-lifter total once per row.
- Scorekeeper accessibility: the platform running-order cards are an `aria-live="polite"` region so "who's up" changes are announced, and the ✓ / ✗ result toggles carry `aria-pressed` so the current result is exposed to assistive tech rather than conveyed by the cell tint alone (the result is also shown by the solid-vs-outline button and the ✓/✗ glyph, so colour is never the sole cue).
- Docs corrected to match the shipped code: CLAUDE.md's attempt-lifecycle rule now describes the scorekeeper's free weight editing and the previous-attempt progression guard (replacing the retired "one increase, no decrease" wording), and the ARCHITECTURE.md §4 real-time map lists `/run` as subscribing to attempts, entries and flights with a note that `referee_decisions` subscriptions are deferred until 3-light refereeing exists.

### Changed
- The competition nav's "Run" link now opens the run screen in its own browser tab (matching the existing new-tab pattern, with the screen-reader "(opens in new tab)" hint), so the full-window scorekeeper runs in a dedicated window during a meet while the admin chrome stays available in the original tab
- Scorekeeper grid (`/run`) no longer compresses its columns to the point of being unreadable on lower-resolution monitors. The table now keeps its designed column widths and scrolls horizontally (`w-full min-w-max`) instead of crushing to fit, and the board now opens in its full-window view by default — the admin chrome otherwise caps content at `max-w-6xl` minus the comp-nav sidebar (~840px), far narrower than a full-power sheet needs. Esc or Collapse returns to the in-flow view, which now scrolls rather than compresses
- Scorekeeper grid (`/run`) touch and legibility tweaks: each attempt's weight is now a full-cell button (min 44px tall, fills the square) so tapping anywhere in the cell opens the weight editor, the ✓ / ✗ result buttons are larger, and all table gridlines are black and slightly thicker (1.5px) for clearer demarcation at the head table
- Scorekeeper grid (`/run`) rows now follow the live running order instead of a static flight-then-lot scoresheet. Within each flight the lifters are ranked by the bar weight of the round in progress (lightest at the top, the order coming down reflecting who lifts next), re-sorting only as each round, then lift, then flight advances; completed attempts keep their slot within the round (the bar weight orders them, not whether they have lifted), so the list stays stable through a round. Lifters whose flight has nothing left to lift fall to the bottom in flight-then-lot order. New pure `orderSessionRoster` helper in `lib/attempts/running-order.ts` (reusing the existing `compareRunningOrder`), unit-tested, consumed by `buildPlatformViews` in `components/scorekeeper/scoresheet-board.tsx`
- Scorekeeper grid (`/run`) is now a denser, full-screen-capable scoresheet. An "Expand to full screen" toggle fills the window over the admin chrome (Esc or Collapse to exit). Each lifter row now shows bodyweight, weight class, division, lot, per-lift rack/bench settings (squat rack height + setting; bench rack/safety heights + spotting), the best successful lift per discipline, and a live running total, with a sticky header and frozen lifter column for wide flights. Weight-class and division names update live from the entries subscription; `bestGoodLift` drives the per-lift best and total. The run page now loads weight classes, divisions and the rack/bodyweight columns to feed the grid. The grid is ruled like a real scoresheet (full gridlines, even fixed-width columns) and a recorded attempt now tints its own cell green (good lift) or red (no lift) instead of carrying a separate label; the lifter currently on the platform is tinted amber. Any attempt's weight is click-to-edit — the scorer can set any value at any time to fix entry errors — and ✓ / ✗ toggle the result (clicking the active one reopens the attempt to pending). The server applies a previous-attempt progression guard (after a good lift the next attempt must be heavier; after a no lift it must be at least the same); first attempts are unconstrained. The old "Raise" box and the one-increase limit are gone
- New `weight_changes` column on `attempts` (migration `20260527000002_attempt_weight_changes.sql`), `types/database.types.ts` hand-updated to match. Added for the original one-increase rule; that rule was superseded by a previous-attempt progression guard, so the column is currently unused (retained, not dropped). Apply the migration via the Supabase SQL editor
- `weighInAction` now mirrors a lifter's openers into their first attempt rows (attempt #1 = the opener) whenever a weigh-in saves, so the scorekeeper's round one is populated straight from weigh-in. Idempotent (re-saving re-syncs attempt #1 to the current opener), seeds only contested lifts (for a team member, just their assigned lift), and is best-effort — a seeding failure is logged to Sentry without failing the weigh-in. Attempts #2 and #3 are created later when declared at the platform (Phase 1 of the live scorekeeping core)
- ARCHITECTURE.md §3 (Permissions) now states explicitly that there is no permissions matrix or `requireRole` API — `requireAdmin()` against `ADMIN_EMAILS` (`lib/auth/admin.ts`) plus the RLS predicates are the whole authorisation model — to prevent future readers hunting for a removed role matrix
- Competition admin pages now share a persistent left-hand navigation sidebar instead of the breadcrumb-style links. A new `CompNav` (`components/comps/comp-nav.tsx`, client, highlights the active section via `usePathname`) and `CompShell` two-column wrapper (`components/comps/comp-shell.tsx`) render in both the new `app/(admin)/[comp-slug]/layout.tsx` (operational pages) and the comp edit page (Setup), which sit in different route segments. The menu lists Setup, Sessions & flights, Lifters, Weigh-in, and — for team comps only — Teams and Team standings (the last opens the public results page in a new tab, since it sits outside the admin chrome). Removed the inline row of `→` links from the comp edit screen and the redundant `← {comp name}` back-links from the entries, weigh-in, flights and teams pages; widened the admin container to `max-w-6xl` to fit the sidebar. The layout and its operational pages share one competition lookup via a request-memoised `getCompBySlug` helper (`lib/comps/get-comp-by-slug.ts`, React `cache()`) rather than each re-fetching the row, and the new-tab Team standings link carries a visually-hidden "(opens in new tab)" hint
- Structured squat/bench rack settings on entries (replacing the two free-text rack-height columns). Squats now store an integer rack height plus a rack setting (IN / OUT / LEFT IN / RIGHT IN); bench stores an integer bench height, an integer safety height, and a spotting choice (SELF / HAND OUT). New `squat_rack_setting` and `bench_spotting` Postgres enums and reworked `entries` columns in migration `20260527000001_weigh_in_rack_settings.sql` (the old text columns are dropped and recreated as integers — no rack data preserved), with `types/database.types.ts` hand-updated to match. Editable on both the weigh-in and entries screens; the `/run` scoresheet (currently a stub) can capture them later once built. Enum labels/values and the rack-height/enum Zod fields added to `lib/constants.ts` and `types/entry.ts` (`entryUpdateSchema` and `weighInSchema`)
- Surnames are now optional for lifters (mononymous athletes and sparse imported data). The first name stays required, so every lifter still has a name; a blank surname is stored as an empty string. Relaxed in the lifter Zod schema, the bulk-import parser, and the add/edit forms. Names now render via a shared `formatLifterName` helper (`lib/lifters/name.ts`) that shows just the first name when the surname is blank, across the entries, flights and teams screens

### Added
- Name filter on the entries screen's "Registered lifters" list (`components/entries/entries-manager.tsx`), matching the weigh-in screen's "Find a lifter…" search: a client-side `type="search"` box that filters the registered entries by lifter name (case-insensitive substring), with a "No lifters match" message when nothing matches. This is the registered-list filter — distinct from the existing "Add a lifter" box, which searches the global lifters database to register someone new

Every edit is optimistic (local state first, rolled back with an inline `role="alert"` message on failure) and reconciles against the attempts/entries real-time subscriptions; attempts are keyed by entry + lift + attempt number so an optimistic write and the realtime insert that follows collapse onto one cell rather than duplicating. `setAttemptWeightAction` returns the attempt's id so the optimistic cell adopts it before that insert arrives. Each runs `adminGuard()`, validates with Zod, is blocked on completed comps via the Phase 0 `canRecordMeetResults` guard, is wrapped in `Sentry.withServerActionInstrumentation`, and verifies the entry/attempt belongs to the named competition (a cross-entity check RLS cannot make). ### Changed
- Simplified the auth model to an admin allowlist (`ADMIN_EMAILS`) plus anonymous public read of published comps. Removed the role-based permission matrix, the `comp_roles` table, the `comp_role` enum, `requireRole()`, the role helpers (`has_comp_role`, `has_any_comp_role`), the creator-grants-meet_director trigger, and the per-comp `overlay_key` and `created_by` columns. Overlays now run on the admin session (no separate overlay auth). See the ADR in ARCHITECTURE.md section 7. Writes rely on `requireAdmin()` in server actions as the sole gate, which requires public sign-ups to stay disabled.
- Public read on `lifters` exposes only non-PII fields: anon reads the `public_lifters` view (name, gender, club, country — no date of birth, no IPF member ID), scoped to lifters in a publicly visible comp; the base table is admin-only.
- Documentation updated to reflect online-only, hosted-first development workflow (no local Next.js or Supabase; migrations applied via the Supabase SQL editor; `types/database.types.ts` hand-authored to match)
- The `@sentry/cli` install build script is currently unapproved (pnpm `onlyBuiltDependencies`). It must be approved before the first production deploy, since Sentry source-map upload runs at build time and depends on the `sentry-cli` binary.
- Email delivery: Supabase default SMTP service used in development until a production domain is registered. Resend integration (per CLAUDE.md and ARCHITECTURE.md) deferred until then. No code changes required at switchover; only Supabase Auth SMTP settings.
