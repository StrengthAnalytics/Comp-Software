# Changelog

All notable changes to Comp-Software are documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/), and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html): `major.minor.patch`.

- **Major**: breaking changes to the data model, public API, or operator-facing workflows
- **Minor**: new functionality, additive schema changes
- **Patch**: bug fixes, copy changes, internal refactors

Update the `[Unreleased]` section in every PR. Cut a new version entry when deploying to production.

### Fixed
- **Names can no longer act as SQL wildcards in the lifter-match lookups.** The approve-submission and bulk-import paths matched an incoming name against existing lifters with `ilike`, where `%`/`_` (and `*`, which PostgREST translates to `%`) are wildcards — so a *public* form submission named `%` would, on approval, have matched an arbitrary existing lifter and overwritten their details. Both lookups now escape the name through a new unit-tested `escapeLikePattern` (`lib/supabase/like-pattern.ts`), so names always match literally; from the pre-merge review.

### Changed
- **The three registration paths now share one set of decision rules.** Single registration (`createEntryAction`), the bulk importer and public-form approval each re-implemented how a date of birth becomes an age category, how a class name resolves to a weight-class row, and how a name finds an existing lifter — three near-copies primed to drift. The rules now live once in `lib/entries/registration.ts` (`deriveAgeCategoryId`, `matchWeightClassByName` — which also now prefers the row for the lifter's own sex when a custom comp gives both sexes the same class name — and `findLifterIdByName`, which owns the wildcard escaping), with unit tests; the three actions keep only their orchestration. Approval also fetches its two reference reads (age categories + weight classes) in parallel instead of sequentially. From the pre-merge `/code-review`; no behaviour change beyond the same-name-both-sexes improvement.
- **Entry form designer: "Open/Close the form" is now a switch.** The accepting-entries control on the Public entry form card is a proper on/off toggle (`role="switch"`, brand-coloured when on) instead of a button, with explanatory text that the form is only live while it's switched on — lifters see "Entries are closed" otherwise. Same optimistic update + rollback behaviour underneath; no action or data change.
- **The comp Overview page is now "Checklist"** — the name describes what the page is for (the setup checklist is its centrepiece). The route moves from `/[comp-slug]/overview` to **`/[comp-slug]/checklist`**, and the sidebar item, comp-switcher landing and comps-list name link all follow. No behaviour change beyond the name and URL.
- **Checklist order: Register lifters now sits above Create platforms & sessions** — lifters are usually entered before the session plan exists, so the checklist teaches that order (`buildSetupChecklist` in `lib/comps/setup-checklist.ts`; the step states, details and links are unchanged, and an order assertion is added to its tests).
- **The comp form's End date now follows the Start date.** Picking a start date defaults an empty (or now-impossible, earlier-than-start) end date to the same day — so opening the end-date picker starts on the comp's month rather than today's, and a one-day meet needs no second entry. An end date already at or after the start is left alone, and the end-date input also gains `min={start}` so the picker can't offer a day before the comp begins (the `ends_on >= starts_on` Zod refinement remains the real gate).

### Fixed
- **Comp slugs can no longer collide with the app's own pages.** A competition slugged `comps`, `records`, `account`, `auth` or `api` would have collided with the matching static route (which wins over `/[comp-slug]` in Next.js), making the comp's public and admin pages unreachable and confusing the sidebar's URL-based comp resolution. ### Fixed
- **UK records no longer appear capped at 1000 — every record now loads.** PostgREST caps a single API response at its `max-rows` setting (1000 by default on Supabase), so the records reads — which fetched the whole table with no pagination — were silently truncated to the first 1000 rows. Records past 1000 *were* saving to the database but never showed on the admin manager (`/records/manage`) or the public browser (`/records`), so adding more looked like a write that wouldn't take. The same cap also hit the bulk importer's existing-key pre-fetch, so with >1000 records on file already-present categories were miscounted as "created" in the import summary (the upsert itself was unaffected — only the reported counts). All three reads now page through the full table via a new shared, unit-tested `fetchAllRows` helper (`lib/supabase/paginate.ts`), which loops `.range()` until the result is exhausted and stays correct even if the server's `max-rows` is set below the page size. Each paged query also carries a final `.order('id')` unique tiebreaker so offset paging can't skip or duplicate a row whose other sort columns tie at a page boundary. The two record pages now read through one shared `loadRecords` loader (`lib/records/load-records.ts`) — parameterised on `includeNotes` so the public page provably never selects the internal `notes` field — which on a read failure returns an empty list and reports to Sentry rather than rendering a silently-truncated partial set as if it were complete. No schema, RLS or data-model change. Tests in `tests/unit/supabase/paginate.test.ts` and `tests/unit/records/load-records.test.ts`.

### Fixed
- **Bulk import maps columns by header name, not fixed position, when the paste includes the header row.** `parseBulkImport` previously read every cell by its ordinal position, so a sheet whose columns were reordered — or whose layout predated a column (e.g. an older export with no "Division" column) — would silently shift each field one place and import wrong weight classes / lots / bodyweights instead of erroring. When a header row is present (first cell "First name"), the parser now derives each column's position from the pasted header labels and aligns by name; a column the header omits reads blank, a column it doesn't recognise is ignored. Pastes without a header fall back to the fixed layout as before. - **UK records — backend foundation (Phase 1 of the regional/national records feature).** Groundwork for an admin-managed, publicly browsable UK powerlifting records dataset (replacing the in-code JSON file of `StrengthAnalytics/BPRecords`). This phase is purely additive and **does not touch the competition data model in any way**: a new standalone `public.records` table with **no foreign keys** to any existing table, two new `record_`-prefixed enums (`record_lift` = squat/bench_press/bench_press_ac/deadlift/total, `record_equipment` = equipped/unequipped) that cannot collide with the eight comp enums, and **RLS policies only on the new table** — no existing table, policy, helper or the `supabase_realtime` publication is altered. - **Standings columns on the run screen and warm-up board** (each togglable from the **Options ▾** dropdown, remembered per browser, default off). For an individual competition: **Current place** (the lifter's live placing within their category, from the running total achieved so far), **Predicted place** (placing from the projected total), **Predicted total** and **Predicted GL points**. The prediction is the GoodLift/LiftingCast "if they make what's declared" projection — each lift's predicted best is the heaviest of that lift's attempts that are either already a good lift *or* still pending with a declared weight (a missed attempt drops out), so a declared-but-untaken third deadlift counts toward the predicted total until it is judged, and a squatter's predicted total already includes their declared bench and deadlift openers. A predicted total is only counted once every contributing lift has a value in play (a bombed lift with nothing pending means no projected total), matching IPF's must-total-in-every-lift rule. Placement category is weight class × division × sex (kit type is fixed per comp); a lifter with no class/division, or no total yet, shows a dash. Ties break by lighter bodyweight then lower lot (the IPF order). For a **team competition** the four individual columns are replaced by **Team points** and **Predicted team points** — the lifter's team's current and projected summed IPF GL (each member's best good lift, or predicted best), shown on every member's row — since a team comp has no individual placing. All values reconcile live through the existing attempts/entries subscriptions (no refresh), and the places rank the whole field comp-wide even though the warm-up board only shows one platform. New pure, unit-tested helpers single-source the maths so the run screen and warm-up board can't disagree: `predictedBestLiftFor`/`computePredictedScore` (`lib/scorekeeper/entry-score.ts`), `computePlacings` (`lib/scorekeeper/placings.ts`) and `computeTeamPoints` (`lib/scorekeeper/team-points.ts`, summing through the shared `teamGlScore`). `BoardEntry` now also carries `weightClassId`, `divisionId` and `teamId` (already fetched by `loadBoardData`) for the category grouping and team aggregation. The weigh-in screen is intentionally unchanged — it has no results yet, only openers being captured.
